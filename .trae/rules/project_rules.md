1. 注释规范：
   - 统一使用中文 //wd 风格，保持简洁明了
   - 关键逻辑、复杂算法、特殊处理必须添加注释
   - 注释应解释"为什么"，而非单纯"是什么"
   - 类、方法、重要变量应有清晰注释

2. 字符串资源管理：
   - 所有新增字符串必须写入 strings_nullgram.xml
   - 英文和中文翻译是必须的，确保国际化支持
   - 字符串命名采用 snake_case 格式，如：message_send_success
   - 避免在代码中直接硬编码字符串

3. 日志规范：
   - 普通日志：android.util.Log.d("wd", "清晰的日志信息")，禁止拼写错误
   - 错误日志：android.util.Log.e("wd", "错误描述", exception)，必须包含堆栈信息
   - 调试日志应在发布前清理或使用条件编译
   - 敏感信息（如用户数据、密钥）禁止写入日志

4. 开发环境配置：
   - 推荐环境：Win + WSL
   - ADB 已映射，直接连接真机调试
   - 确保 Android SDK 路径正确配置在 local.properties 中

5. 任务完成验证流程：
   - 必须运行：./gradlew :TMessagesProj:lintDebug 检查代码质量
   - 修复所有 lint 错误，警告根据情况处理
   - 电脑配置不足时，禁止直接执行编译运行验证
   - 仅在明确要求时执行：./gradlew :TMessagesProj:assembleDebug 安装到真机
   - 代码提交前必须确保本地编译通过

6. 文档查阅规范：
   - 优先查阅 TGwiki 定位实现细节
   - 再对照官方 Telegram API 文档确认字段与行为
   - 遇到问题时，先搜索项目已有代码和提交历史

7. 代码风格：
   - 遵循 Android Studio 默认的代码格式化规则
   - 缩进使用 4 个空格，禁止使用 Tab
   - 每行代码长度不超过 120 个字符
   - 方法参数较多时，每行一个参数，提高可读性

8. 依赖管理：
   - 新增依赖必须经过验证，避免引入安全风险
   - 优先使用项目已有的依赖版本
   - 定期检查并更新依赖，修复安全漏洞

9. 版本控制：
   - 提交信息应清晰描述修改内容，格式：[模块] 简要描述
   - 避免大文件提交，使用 .gitignore 排除不必要的文件
   - 定期拉取最新代码，避免冲突

10. 性能优化：
    - 避免在主线程执行耗时操作
    - 合理使用缓存，减少重复计算
    - 图片加载应使用异步加载和压缩处理
    - 内存管理：及时释放不再使用的资源

11. 安全规范：
    - 敏感数据必须加密存储
    - 网络请求必须使用 HTTPS
    - 避免 SQL 注入、XSS 等安全漏洞
    - 权限申请应遵循最小必要原则
